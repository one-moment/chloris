# 협업 시스템 — boro-flower-market × chloris

> 이 문서는 팀 4명(리더 · 기획 · 디자인 · 개발)이 **공통으로 공유하는 운영 매뉴얼**입니다.
> 모든 레포의 `docs/`에 두고, 각 `CLAUDE.md`/지침이 이 문서를 기준으로 삼습니다.

## 1. 팀과 역할

| 역할 | 누구 | 한다 | 안 한다 |
| --- | --- | --- | --- |
| 프로젝트 리더 | 사용자 | 방향 결정 · 최종 승인 · 에이전트 간 전달 | — |
| 기획자 | claude.ai의 Claude | 기획 · 구조 설계 · 작업지시서 · 검토 | 직접 구현/디자인 |
| UX 디자이너 | Claude Design | 디자인 시스템 · 브랜드 테마 · 화면 시안 | 코드 구현 |
| 개발자 | Claude Code | 코드 구현 · 테스트 · 커밋 | 범위 밖 변경 |

## 2. 핵심 전제 — 에이전트는 서로 직접 대화하지 않는다
- 기획 · 디자인 · 개발 세 에이전트는 실시간으로 연결되지 않습니다.
- 모든 연결은 **GitHub 문서 + 리더의 전달**로 이뤄집니다.
- 따라서 **문서가 곧 의사소통**입니다. 결정 · 요청 · 산출물은 전부 문서로 남깁니다.

## 3. 단일 기준 문서 (authoritative)
- 기획: `docs/chloris-기획정리.md`
- 디자인: `docs/design-system.md`(공통) + `docs/brand-theme-boro.md`(보로 테마) ← Claude Design 산출
- 개발 규칙: `CLAUDE.md`
- 팀 운영: 이 문서 `docs/협업시스템.md`
- **충돌 시 우선순위:** 리더 결정 > 기획정리 > 디자인 시스템 > 개별 작업지시서.
  같은 주제가 두 문서에서 다르면 위쪽이 이깁니다. 발견하면 리더에게 보고하세요.

## 4. 산출물 흐름 (한 방향)
```
리더 요청
  → [기획] 작업지시서 · 구조
    → [디자인] 화면 · 컴포넌트 시안 + 디자인 토큰
      → [개발] 구현 · 테스트
        → 리더 검수
```
- 각 단계는 **리더 승인 후** 다음으로 넘어갑니다.
- **디자인↔개발 정합:** 디자인 토큰(색 등)의 이름은 개발의 `brand.config` 값과 맞춥니다. 그래야 시안이 그대로 구현됩니다.

## 5. 작업 라벨 (모든 작업·산출물 앞에)
- `[코어]` — 공통 (chloris-core)
- `[브랜드:보로]` — 보로 전용
- `[디자인:공통]` / `[디자인:보로]` — 디자인 작업
예) `[디자인:보로] 주문 카드 컴포넌트 시안`

## 6. 공통 규칙 (모두 적용)
- 절차: 계획 먼저 → 리더 승인 → 작은 단위 → 검토/테스트 → (개발은) 한국어 커밋
- 금지: 파일/DB 삭제·초기화, `git push --force`/`git reset --hard`, 프로덕션 배포, 범위 밖 변경, 비밀키 코드에 직접 작성
- 소통: 비개발자가 이해할 쉬운 한국어 / 작업 후 "무엇을 · 어떤 위험 · 확인법" 요약 / 불확실하면 질문

## 7. 핸드오프 규칙 (디자인 → 개발)
디자인이 개발로 넘길 때 함께 명시합니다:
- 컴포넌트 이름 · 상태(기본/호버/포커스/비활성/에러)
- 간격·타이포 토큰 · 색 토큰 이름(= `brand.config`와 일치)
- 반응형 동작 · 접근성 요건(대비·키보드·라벨)

**디자인 산출물 기록 규칙 (중요):** 디자인 산출물은 반드시 `docs/`에 **글로 요약**하고, **원본 링크를 함께 적습니다.**
- 요약 위치: `docs/design-system.md`(공통) · `docs/brand-theme-boro.md`(보로 테마)
- 요약 내용: 컴포넌트 · 상태 · 토큰 이름, 색 값, 그렇게 정한 이유
- 외부 핸드오프 링크(예: `api.anthropic.com/v1/design/...`)는 "원본" 표시와 함께 기록만 합니다. 링크만 넘기지 마세요.
- **링크는 지름길, 문서는 기준.** 둘이 다르면 문서가 이깁니다.
- 이유: 기획·리더는 외부 링크를 열 수 없고, 링크는 기록이 아니라 전달일 뿐입니다. GitHub 문서로 남아야 팀 전체가 진행을 검토·관리할 수 있습니다.

## 8. 보로 특수사항
- 보로는 가장 앞선 레포입니다 → 잘 나온 공통 패턴은 `[코어]` / `[디자인:공통]` 후보로 승격합니다.
- 기능개선 빌드가 진행 중입니다 → 구조·디자인 정리는 **별도 브랜치/별도 산출물**로, 진행 중 작업과 섞지 않습니다.

## 9. 기획 트랙 구조 (누가 어디서 기획하나)
코드 구조(코어 + 브랜드 3개)와 똑같은 모양으로 기획도 나눕니다.
- **코어·플랫폼 기획 트랙 1개** — `chloris-core`, 협업 시스템, 공통 규칙, 브랜드 간 결정을 단독으로 소유합니다.
- **브랜드 기획 트랙 3개** — 각자 자기 브랜드의 커스텀·CRM에 집중합니다. 코어 결정은 "주어진 것"으로 받아 씁니다.
- **올림(escalation) 규칙:** 브랜드 트랙에서 코어를 바꿔야 할 일이 생기면 거기서 정하지 말고 코어 트랙(+리더)으로 올립니다. (코드의 `[코어]` 라벨 규칙과 동일)
- **진짜 기준은 항상 GitHub 문서.** 기획 트랙을 나눠도, 모두 단일 기준 문서(3번)에서 출발하고 문서로 남기면 정합성이 유지됩니다.
